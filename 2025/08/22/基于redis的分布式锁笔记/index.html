背景

项目从单机转为集群时，服务会多地多分运行。如何保证定时任务只有一个服务去执行，这儿可能会有小伙伴说使用 xxl-job 做分布式定时任务，为下次更新挖个坑！这儿咱先只看redis分布式锁

思路

[!warning]
在 Redis 2.6.12 版本以前， SET 命令总是返回 OK 。
从 Redis 2.6.12 版本开始， SET 在设置操作成功完成时，才返回 OK 。

主要基于 redis 的 SET key value [EX seconds] [PX milliseconds] [NX|XX] 命令，官方 文档解释如下

此命令有两个可选项EX和NX
EX 秒 ——设置指定的过期时间，以秒为单位（正整数）。
NX 仅当密钥不存在时才设置它。

为基于redis 实现分布式锁奠定了基础，NX可以保证只有一个key存在然后通过EX设置的过期时间自动释放锁，在redisTemplate中redisTemplate.opsForValue().setIfAbsent(key, value, timeout, unit) 等价于 Redis 的 SET NX EX，这儿NX 对应 setIfAbsent，EX对应setIfAbsent方法中的 timeout 参数。

那么我该如何手动释放这个锁呢？
第一反应就是redisTemplate.delete(key),但这个操作并不是原子操作，正确的做法是使用Lua脚本保证原子性。

解锁实现

现在加锁/解锁都有了，定时任务拿到锁之后在超时时间内无法完成任务。此时就需要我们进行续锁,最开始直接一个while循环增加时间，调用处对autoLockThread调用interrupt()中断线程，下次自动续锁的时候Thread.sleep() 就会抛出 InterruptedException 从而中断自动续锁，注意：这儿并不会直接释放锁。

使用ScheduledExecutorService 优化上面通过循环去续锁，它提供了 定时任务调度 的能力。其中的scheduleAtFixedRate() 有一段秒速是这样的 “==如果此任务的任何执行耗时超过其周期，则后续执行可能会延迟开始，但不会并发执行。==” 就可以保证每个定时任务执行完才会释放锁。

缺点

对于只确保某一个服务去执行定时任务，当前实现的锁已能满足需求。但它的设计缺少了‘可重入性’，这使得当任务逻辑变得复杂、代码需要模块化拆分时，它会暴露出设计上的短板，甚至导致死锁。因此，从构建一个通用、健壮的分布式锁组件的角度来看，不支持可重-入是 一个个重要的缺陷。同时看门狗线程创建和维护也是一个问题，复杂环境中更推荐使用Redisson这样成熟的解决方案。它不仅解决了以上所有问题，还提供了更优雅的 API 和更丰富的功能：”

  • 可重入锁：Redisson 的 RLock 默认就是可重入的。
  • 简洁的 API：它的接口与java.util.concurrent.locks.Lock完全一致，使用 lock.lock() 和 lock.unlock() 即可，内置的看门狗会自动启动和停止，用户无感知。
  • 健壮的看门狗：Redisson 内部对线程池和资源管理做了大量优化。
  • 支持公平锁：提供了 getFairLock() 方法来获取公平锁。
  • 丰富的功能：内置了读写锁、联锁、信号量、闭锁（CountDownLatch）等多种分布式同步工具。
